# Feature: Registry Framework <!-- rq-250be957 -->

Every open-extensible builder collection in the engine is a `Registry<B>`:
a host-side container of boxed builder trait objects that an embedder can
extend by registering additional builders before a simulation runs. The
engine has seven registries — integrators, thermostats, barostats,
constraint types, potentials, minimizers, and analyses — bundled together
in `Registries` (see `simulation-runner.md`). All seven share one generic
container type; the differences between them are confined to which builder
trait they hold, which built-ins they carry, and how the rest of the code
dispatches construction.

The framework distinguishes two selection models, and a registry belongs to
exactly one of them:

- **Named selection.** The configuration names a `kind` string; the registry
  looks up the single builder whose `kind_name()` matches and constructs from
  it. Used where the choice is a user decision among mutually-exclusive
  alternatives: the integrator, thermostat, barostat, and minimizer
  registries (one selection per phase), and the constraint-type and analysis
  registries (several instances, each still naming its kind). A named-selection
  registry's builder trait carries a kind key.
- **Compositional activation.** The configuration carries no kind for the
  slot; each builder inspects the build context and returns `Some(slot)` when
  the data it consumes is present and `None` otherwise. The result is the set
  of all builders that activate, summed together. Used by the potential
  registry alone: a force field is an additive composition of many
  simultaneously-active potentials, each switched on by a distinct
  configuration section (Lennard-Jones by `[[pair_interactions]]`, SPME by
  `[spme]`, Morse by the bond list, harmonic angles by the angle list,
  periodic dihedrals by the dihedral list). The potential builder trait
  carries no kind key.

The generic container provides storage, registration, built-in population,
cloning, and — for named-selection registries only — lookup by kind. The
per-subsystem construction dispatch (selecting and building from a parsed
slot, building the optional thermostat/barostat, or iterating every potential
builder for activation) is not part of the container; each subsystem owns its
own dispatch because the build inputs differ by subsystem.

## Builder Trait Layering <!-- rq-87d01d67 -->

A builder trait is the interface a subsystem's factories implement (for
example `IntegratorBuilder`, `PotentialBuilder`). Every builder trait is
`Debug + Send + Sync` and supports being cloned as a boxed trait object, so
that a `Registry` holding it is itself `Clone`. Boxed-trait-object cloning is
generated by the `registry_builder_clone!` macro, which emits a per-trait
clone-helper supertrait (`<Trait>Clone`), a blanket impl over any `Clone`
builder, and `impl Clone for Box<dyn Trait>`. A concrete builder needs only
`#[derive(Clone)]` and writes no per-builder clone method. The helper trait
is non-generic — its trait-object type appears only in the method return,
never as a trait type parameter — which is what lets it serve as a supertrait
without the super-predicate cycle a `Trait: CloneToBox<dyn Trait>` bound would
induce.

A named-selection builder trait additionally implements `KindedBuilder`,
which supplies the `kind_name()` lookup key. The six named-selection builder
traits carry this supertrait bound; `PotentialBuilder` does not. Because the
keyed `lookup` method is available only when the held builder is a
`KindedBuilder`, `PotentialRegistry` has no `lookup` method at all — the
distinction between the two selection models is enforced by the type system,
not by convention.

## Built-in Rosters <!-- rq-55fd6f39 -->

Each registry's built-in roster — the list of builders
`with_builtins()` pre-populates — is the one genuinely per-registry value.
A builder trait that has a default roster implements `Builtins`, whose
`builtins()` returns the boxed built-in builders in canonical registration
order. `Registry::with_builtins()` and the `Default` impl are generic over
`Builtins`, so the roster is the only per-subsystem code the population path
needs.

## Concrete Registries <!-- rq-223901a4 -->

Each concrete registry is a type alias for the generic container specialised
to one builder trait object:

```rust
pub type IntegratorRegistry  = Registry<dyn IntegratorBuilder>;
pub type ThermostatRegistry  = Registry<dyn ThermostatBuilder>;
pub type BarostatRegistry    = Registry<dyn BarostatBuilder>;
pub type ConstraintRegistry  = Registry<dyn ConstraintBuilder>;
pub type MinimizerRegistry   = Registry<dyn MinimizerBuilder>;
pub type AnalysisRegistry    = Registry<dyn AnalysisBuilder>;
pub type PotentialRegistry   = Registry<dyn PotentialBuilder>;
```

`XRegistry::with_builtins()`, `::new()`, `.register(...)`, and (for the six
named-selection registries) `.lookup(kind)` resolve through the alias to the
generic container's methods. A subsystem's construction dispatch lives in an
inherent `impl Registry<dyn XBuilder>` block in that subsystem's module
(for example `Registry<dyn IntegratorBuilder>::build`, which looks the builder
up by `slot.kind` and constructs it); these dispatch methods are documented in
the per-subsystem framework files and are not part of this framework.

## Feature API <!-- rq-54fc86d6 -->

### Types <!-- rq-d9344e8e -->

- `Registry<B: ?Sized>` — generic host-side container of boxed builders. <!-- rq-e0ea3802 -->

  ```rust
  pub struct Registry<B: ?Sized> {
      builders: Vec<Box<B>>,
  }
  ```

  - `new() -> Self` — an empty registry (no builders).
  - `from_builders(builders: Vec<Box<B>>) -> Self` — a registry holding the
    given builders in the given order.
  - `register(&mut self, builder: Box<B>)` — appends `builder`, preserving
    registration order.
  - `builders(&self) -> &[Box<B>]` — the held builders in registration order,
    for subsystems that iterate every builder (compositional activation).
  - `with_builtins() -> Self` — available when `B: Builtins`; constructs a
    registry pre-populated with `B::builtins()` in canonical order.
  - `lookup(&self, kind: &str) -> Option<&B>` — available when
    `B: KindedBuilder`; returns the first held builder whose `kind_name()`
    equals `kind` in registration order, or `None` if no builder matches.
  - `Clone` — available when `Box<B>: Clone` (every builder trait gets this
    via `registry_builder_clone!`); clones the registry by cloning each
    boxed builder.
  - `Default` — available when `B: Builtins`; equals `with_builtins()`.
  - `Debug` — derived; available when `B: Debug`.

- `KindedBuilder` — the keyed-lookup capability a named-selection builder <!-- rq-0f6b7b7a -->
  trait carries, plus ownership of its kind's unit conversion.

  ```rust
  pub trait KindedBuilder {
      fn kind_name(&self) -> &'static str;

      /// Rescale this kind's open-shaped `params` table from the user's
      /// unit system to atomic units, in place. The implementation
      /// deserialises `params` into the builder's typed parameter
      /// struct (which derives `Convert`; see `io/unit-system.md`),
      /// filling any omitted field with its serde default, applies
      /// `Convert::from_user`, and serialises the full converted struct
      /// back into `params` — every field, defaulted or user-supplied,
      /// so an omitted dimensioned default is rescaled like a
      /// user-written value (see `io/unit-system.md`). A non-numeric
      /// value at a unit-bearing field surfaces the typed
      /// deserialisation error. Default: a no-op, appropriate only for a
      /// kind with no unit-bearing params.
      fn convert_params(
          &self,
          _units: UnitSystem,
          _params: &mut toml::Value,
      ) -> Result<(), ConfigError> {
          Ok(())
      }
  }
  ```

  The six named-selection builder traits (`IntegratorBuilder`,
  `ThermostatBuilder`, `BarostatBuilder`, `ConstraintBuilder`,
  `MinimizerBuilder`, `AnalysisBuilder`) carry `KindedBuilder` as a
  supertrait. `PotentialBuilder` does not, which is what withholds `lookup`
  from `PotentialRegistry`; pair/bonded potential params are typed
  `Config` fields converted by the loader's single `Convert` pass rather
  than per-builder. Because `convert_params` is co-located with the
  builder that owns the kind's schema and is generated from the typed
  params struct's dimensioned fields, a built-in kind cannot be
  registered without its params conversion being defined.

- `Builtins` — the per-trait built-in roster. <!-- rq-c00689e6 -->

  ```rust
  pub trait Builtins {
      fn builtins() -> Vec<Box<Self>>;
  }
  ```

  Implemented once per builder trait object (for example
  `impl Builtins for dyn IntegratorBuilder`), returning the boxed built-in
  builders in canonical registration order.

- `registry_builder_clone!($vis $helper for $trait)` — macro that makes <!-- rq-b775df32 -->
  `Registry<dyn $trait>` cloneable without a per-builder clone method. It
  expands to a clone-helper supertrait, a blanket impl, and the boxed-trait
  `Clone` impl:

  ```rust
  registry_builder_clone!(pub IntegratorBuilderClone for IntegratorBuilder);
  // expands to:
  pub trait IntegratorBuilderClone {
      fn registry_clone_box(&self) -> Box<dyn IntegratorBuilder>;
  }
  impl<T: IntegratorBuilder + Clone + 'static> IntegratorBuilderClone for T {
      fn registry_clone_box(&self) -> Box<dyn IntegratorBuilder> { Box::new(self.clone()) }
  }
  impl Clone for Box<dyn IntegratorBuilder> { /* calls registry_clone_box */ }
  ```

  Each builder trait carries its generated `<Trait>Clone` helper as a
  supertrait bound; the helper is non-generic (the trait-object type is only
  in the method return), so it serves as a supertrait without a
  super-predicate cycle. No builder trait declares an explicit `box_clone`
  method, and no concrete builder writes one — `#[derive(Clone)]` suffices.

- Concrete registry type aliases — `IntegratorRegistry`, `ThermostatRegistry`,
  `BarostatRegistry`, `ConstraintRegistry`, `MinimizerRegistry`,
  `AnalysisRegistry`, `PotentialRegistry`, each `Registry<dyn XBuilder>` for
  the corresponding builder trait. Re-exported from their subsystem modules.

### Behaviours <!-- rq-24fb4604 -->

- `lookup` matches on `kind_name()` equality and returns the **first** match <!-- rq-cb2f6a59 -->
  in registration order. A builder registered after a built-in of the same
  kind is therefore shadowed by the built-in; registration does not override
  an existing kind.
- `register` never deduplicates: two builders of the same kind may both be <!-- rq-b3e0f901 -->
  held, and `lookup` returns the earlier one.
- Cloning a registry is deep: the clone holds independently-boxed copies of
  every builder, so mutating one registry's builder list does not affect a
  clone.
- A `Registry<dyn PotentialBuilder>` value exposes `new`, `from_builders`,
  `register`, `builders`, `with_builtins`, `Clone`, and `Default`, but not
  `lookup`. Attempting to call `lookup` on it is a compile error.

## Out of Scope <!-- rq-c4300389 -->

- The per-subsystem construction dispatch (`Registry<dyn IntegratorBuilder>::build`,
  `Registry<dyn ThermostatBuilder>::build_optional`, the potential-activation
  loop in `ForceField::new`, the constraint-group partitioning in the
  constraint registry, and so on). These take subsystem-specific build inputs
  and live in the subsystem modules; see the per-subsystem framework files.
- The `Registries` bundle that aggregates the seven concrete registries and
  offers `register_*` convenience methods; it is documented in
  `simulation-runner.md`.
- Any change to the configuration's selection model. Potentials remain
  keyless (compositional activation); the named-selection registries remain
  keyed by `kind`. Giving potentials a `kind` key, or letting a registered
  builder override a built-in of the same kind, are deliberate non-goals.
