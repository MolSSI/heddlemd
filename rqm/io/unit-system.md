# Feature: Unit-System Selector and Atomic-Units Internals <!-- rq-a570f6d2 -->

The simulation engine stores and computes in **Hartree atomic units**
internally. The TOML configuration file, the extended-XYZ initial-state
file, and the engine's output files (trajectory, CSV log,
minimization log) all support two unit systems for I/O: SI (the default)
and Hartree atomic units. A single top-level TOML field, `units`,
selects which system every I/O file attached to a given run is written
in; the loader converts on read and the writers convert on write so the
user's view of the system is consistent end-to-end.

The atomic-unit choice eliminates the universal physical-constant
prefactors that otherwise complicate the engine's kernels:
**`k_B = 1`** (no Boltzmann constant appears in any thermostat or
temperature formula), **`1 / (4πε₀) = 1`** (no Coulomb prefactor
appears in any pair-force or SPME kernel), and the dynamic range of
every physical scalar lands close to order unity, comfortably inside
`f32`.

## Supported Unit Systems <!-- rq-0eeaeb92 -->

Two values are accepted for the top-level `units` field:

- `"si"` (default) — every numeric scalar in the TOML config, every
  numeric column in the referenced `.in.xyz` file, and every row in
  every output file the run produces is expressed in the SI system:
  length in metres, mass in kilograms, time in seconds, energy in
  joules, temperature in kelvin, pressure in pascals, charge in
  coulombs, velocity in metres per second.
- `"atomic"` — values are expressed in Hartree atomic units. Length is
  in Bohr radii (`a_0`), mass in electron rest masses (`m_e`), charge
  in elementary charges (`e`), energy in Hartrees (`E_h`), time in
  atomic time units (`hbar / E_h`), pressure in `E_h / a_0^3`,
  temperature in `E_h / k_B` (i.e. each "temperature" value is the
  numerical equivalent of `k_B · T` in Hartrees), and velocity in
  `a_0 / (hbar / E_h)`.

Comparison is exact and case-sensitive: `"Atomic"`, `"ATOMIC"`,
`"SI"`, and any other casing or value yield
`ConfigError::UnknownUnits { got: <value> }`. Omitting the field is
equivalent to `units = "si"`.

The chosen unit system is uniform across every file attached to a
single run: the TOML config, the `.in.xyz` initial state it
references, and the trajectory/log/minlog files the runner produces
under that config's `[phase.output]` and `[minimization.output]`
blocks all share the same selector. A single `Config.units: UnitSystem`
value is threaded to the relevant readers and writers.

## Physical Constants Source <!-- rq-2be0d376 -->

Atomic-unit conversion factors derive from the **CODATA-2018**
recommended values for the fundamental physical constants, as exported
by the QCElemental project. The exact decimal values used at build time
live in `src/units/atomic_constants.rs`, which is generated by
`scripts/gen_atomic_units.py`. Any QCElemental release that ships the
CODATA-2018 physical-constants context produces an acceptable file;
the generator script does not pin a QCElemental version.

The generated file is committed to the repository so that builds do not
depend on a Python toolchain. Regenerating the file after a future
CODATA release (e.g. CODATA-2022) is the mechanism by which the engine
adopts updated constants.

## Internal Representation <!-- rq-1be48dbd -->

The engine's host-side and device-side state is in atomic units
throughout. Concretely:

- `ParticleBuffers` (positions, velocities, forces, masses, charges)
  carries Bohr radii, atomic-unit velocities, atomic-unit forces,
  electron masses, and elementary charges.
- `SimulationBox` carries lattice parameters in Bohr.
- Every `[phase]` `dt` and slot-parameter `tau` / `tau_t` / `tau_p`
  carries atomic time units after deserialisation.
- Every "temperature" the engine handles (Maxwell-Boltzmann initial
  velocities, thermostat setpoints, the per-step `temperature` log
  column produced by `compute_temperature`) carries Hartrees and is
  numerically identical to `k_B · T` in atomic units.
- Every "pressure" the engine handles (barostat setpoints, the
  instantaneous virial pressure) carries `E_h / a_0^3`. Every
  "compressibility" carries `a_0^3 / E_h`.

The two universal physical-constant prefactors take their unit values:

- **Boltzmann constant `k_B = 1` exactly.** No `k_B` constant appears
  in the kinetic-energy → temperature relation, in Maxwell-Boltzmann
  initial-velocity sampling, in any thermostat's drag/diffusion
  coefficients, or in the barostat-virial coupling. The relation
  `T = 2 · K / N_thermal_dof` holds directly with all three quantities
  in atomic units.
- **Coulomb prefactor `1 / (4πε₀) = 1` exactly.** The truncated
  short-range Coulomb pair force and the SPME real-/reciprocal-space
  kernels carry no permittivity constant; the force on particle `i`
  from particle `j` is `F = q_i · q_j · r̂ / r²` (with the appropriate
  switching window and Ewald split applied as documented in
  `coulomb-pair-force.md` and `spme.md`).

Wall-clock timing data (kernel-stage durations recorded in the
`.timings` file, runner-side host timers) remains in SI seconds /
milliseconds. Wall-clock time is a profiling channel for measuring GPU
and host work; it is not a physical quantity and the `units` selector
does not apply to it.

## Conversion Semantics <!-- rq-a2d8b353 -->

The conversion is a single floating-point multiplication per scalar,
applied at the I/O boundary in the direction appropriate to the file
side:

```
value_atomic = value_user / factor(dimension)    # input  (user → engine)
value_user   = value_atomic * factor(dimension)  # output (engine → user)
```

`factor(dimension)` is the SI value of one atomic unit of the named
dimension. For `UnitSystem::Si` the conversion is a non-trivial
multiplication; for `UnitSystem::Atomic` the multiplier reduces to
`1.0` (since the user's view and the engine's view coincide) and the
conversion is a numerical identity. Inverse dimensions
(`InverseLength`, `InverseTime`, `InversePressure`) are derived in
software as `1.0 / factor(base_dimension)` rather than stored
independently.

Input-side conversion targets:

- Every unit-bearing scalar in the loaded `Config` (typed fields under
  `[simulation]`, `[[phase]]`, `[[particle_types]]`,
  `[[pair_interactions]]`, `[[bond_types]]`, `[[angle_types]]`,
  `[coulomb]`, `[spme]`, `[neighbor_list]`).
- The unit-bearing fields of every open-shaped slot's
  `params: toml::Value` whose `kind` is recognised by the unit module
  (`[phase.integrator]`, `[phase.thermostat]`, `[phase.barostat]`,
  `[[constraint_types]]`, `[minimization.algorithm]`).
- The `Lattice` attribute, position columns, and velocity columns of
  the `.in.xyz` initial-state file passed to `load_init_state`.

Output-side conversion targets:

- The `Lattice` attribute, position columns, and velocity columns of
  every trajectory frame written by `TrajectoryWriter::write_frame`.
- The `time`, `kinetic_energy`, `temperature`, and any integrator-
  declared extra-diagnostic columns written by `LogWriter::write_row`.
- The `time`, `potential_energy`, `max_force`, `step_size`, and any
  algorithm-declared extra-diagnostic columns written by the
  minimization log (`MinlogWriter::write_row` in
  `rqm/minimization/steepest-descent.md`).

Each writer is constructed with a `UnitSystem` chosen by the caller
(`config.units` for trajectory and CSV log writers attached to a
run); the writer applies the output direction of the conversion as
each row is formatted. The columns themselves are still bit-exact
reproducible: identical inputs produce identical formatted output
bytes.

Defaults the loader computes from other fields (e.g.
`r_switch = 0.9 * cutoff`, `r_skin = 0.3 * max_cutoff`) inherit their
dimension from the field they default off; the dimensionless ratio is
multiplied by an already-converted atomic-unit value, so the default
ends up atomic without any additional conversion logic.

Validation runs on the post-conversion (atomic-unit) values. Range
checks (`require_finite_positive`, `r_switch <= cutoff`,
`r_hh < 2 * r_oh`, etc.) are unit-agnostic — they depend only on sign,
finiteness, or ordering, all of which are preserved by division by a
strictly positive factor — so the same validation logic catches the
same errors regardless of which unit system the user wrote.

### Per-slot field-to-dimension mapping <!-- rq-bf5df23e -->

The unit module owns a static table mapping each known slot `kind`
string to the list of `(field_name, Dimension)` pairs that carry
units. The table covers every built-in integrator, thermostat,
barostat, constraint, and minimizer registered in the default
registries. When a slot's `kind` is recognised, the listed fields are
rescaled and every other field of `params` is left untouched. When the
`kind` is not in the table the entire `params` table is left untouched;
the kind itself is rejected later by
`Config::validate_against(&registries)` with
`ConfigError::UnknownKind`.

Non-numeric values at a known unit-bearing field name (e.g. a string
where the loader expects a temperature) are left untouched. The
builder's `validate_params` then produces the appropriate
type-mismatch error in its native flow.

## Initial-State File <!-- rq-be580c3c -->

`load_init_state` takes a `UnitSystem` parameter and applies the
input-side conversion to the file's numerical columns: the nine
`Lattice` components are divided by the length factor before the
`SimulationBox` is constructed; the position columns are divided by
the length factor; the velocity columns are divided by the velocity
factor. The position-in-box check runs against the converted
`SimulationBox`, so positions and box edges are consistently in Bohr
inside the returned `InitState` whichever system the file is written
in.

A caller typically threads `Config.units` through to
`load_init_state` so that the `.in.xyz` file is interpreted in the
same system as the TOML config that references it; the loader itself
does not enforce this pairing.

## Output Writers <!-- rq-d50cd03c -->

`TrajectoryWriter`, `LogWriter`, and the minimization-log writer each
accept a `UnitSystem` at open time and apply the output-side
conversion as each frame or row is formatted. When the `UnitSystem`
is `Si`, every f64 column passes through a multiplication by the SI
value of one atomic unit (lengths by the bohr → metre factor; times
by the atomic-time → second factor; energies by the hartree → joule
factor; temperatures by the (E_h / k_B) → kelvin factor; etc.). When
the `UnitSystem` is `Atomic`, the multiplier is `1.0` for every
column and the writer emits the engine's internal scalars directly.

The runner passes `config.units` to every output-writer constructor it
creates for the phases of that config, so a single `units = "atomic"`
selector at the top of a TOML file produces atomic-unit input *and*
atomic-unit output across the whole simulation; a single
`units = "si"` selector produces SI-unit input *and* SI-unit output.
Mixing is not supported.

## Feature API <!-- rq-f5984310 -->

### Types <!-- rq-6fec4fdb -->

- `UnitSystem` — discriminated enum naming the two supported systems. <!-- rq-34446ef5 -->

  ```rust
  pub enum UnitSystem {
      Si,
      Atomic,
  }
  ```

  Implements `Default` with `Si` as the default variant. The string
  form used by the TOML schema is `"si"` for `Si` and `"atomic"` for
  `Atomic`; the `from_str` / `as_str` helpers round-trip those two
  forms and return `None` / fail for anything else.

- `Dimension` — discriminated enum naming the physical dimensions the <!-- rq-0d49f455 -->
  conversion table recognises:

  ```rust
  pub enum Dimension {
      Dimensionless,
      Length,
      InverseLength,
      Mass,
      Charge,
      Energy,
      Time,
      InverseTime,
      Force,
      Pressure,
      InversePressure,
      Temperature,
      Velocity,
  }
  ```

  Every unit-bearing scalar the loader or writer handles maps to
  exactly one of these variants. `Dimensionless` is the identity for
  both `to_si` and `from_si` regardless of `UnitSystem`; it tags
  counter-like or already-normalized values (e.g. an integrator
  diagnostic that reports a pure ratio) that flow through the
  conversion pipeline unchanged.

### Functions <!-- rq-1669abef -->

- `UnitSystem::factor(self, dim: Dimension) -> f64` <!-- rq-fb7c4429 -->
  - Returns the SI value of one atomic unit of the named dimension,
    independent of `self`'s direction.
  - For `UnitSystem::Atomic`, returns `1.0` for every dimension
    (the user's view and the engine's view coincide).
  - For `UnitSystem::Si`, returns the SI-per-atomic-unit factor drawn
    from `src/units/atomic_constants.rs` (CODATA-2018 via
    QCElemental).
  - Inverse dimensions return the reciprocal of the corresponding base
    dimension's factor.

- `UnitSystem::to_si(self, dim: Dimension, value: f64) -> f64` <!-- rq-a7677c61 -->
  - Output-direction convenience: returns `value * self.factor(dim)`.
    Used by the trajectory, CSV log, and minlog writers to translate
    one engine-side atomic-unit scalar into the user's chosen unit
    system.

- `UnitSystem::from_si(self, dim: Dimension, value: f64) -> f64` <!-- rq-fdeba84b -->
  - Input-direction convenience: returns `value / self.factor(dim)`.
    Used by `load_config` and `load_init_state` to translate one
    user-supplied scalar into the engine's atomic-unit representation.

- `UnitSystem::from_str(s: &str) -> Option<Self>` <!-- rq-1c857adf -->
  - Exact-case lookup: returns `Some(Si)` for `"si"`, `Some(Atomic)`
    for `"atomic"`, and `None` for every other input.

- `UnitSystem::as_str(self) -> &'static str` <!-- rq-7513f933 -->
  - Returns `"si"` or `"atomic"`, the canonical lowercase string
    representation. Round-trips with `from_str`.

- `slot_kind_field_dims(kind: &str) -> Option<&'static [(&'static str, Dimension)]>` <!-- rq-ecb8aa60 -->
  - Returns the list of unit-bearing fields for a recognised slot
    `kind`, or `None` for an unrecognised kind.
  - A recognised kind with no unit-bearing fields (e.g.
    `"velocity-verlet"`) returns `Some(&[])`. The presence of the
    entry is itself the contract that the kind was considered.

- `convert_slot_params(system: UnitSystem, kind: &str, params: &mut toml::Value)` <!-- rq-8f5ebdc1 -->
  - In-place rescales every unit-bearing field of an open-shaped slot's
    `params` table to atomic units (i.e. applies the input direction).
  - No-op when `system == UnitSystem::Atomic`.
  - No-op when `kind` is not present in `slot_kind_field_dims`.
  - For each `(field, dimension)` returned by `slot_kind_field_dims`,
    divides the field's numeric value by `system.factor(dimension)`.
    Integer-typed values are promoted to floats during the conversion;
    non-numeric values are left untouched.

## Out of Scope <!-- rq-f033db03 -->

- **Mixed unit systems within a single run.** A configuration with
  some atomic-unit fields and some SI-unit fields is not supported;
  one selector applies uniformly to the entire TOML file, the
  referenced `.in.xyz` file, and every output file the run produces.
- **Per-quantity unit suffixes.** Values like `cutoff = "1.0 nm"` or
  `mass = "16.0 amu"` are not parsed; values are bare floats
  interpreted under the file-level `units` selector.
- **Reduced-unit systems** (LJ-reduced, GROMACS nm/ps, LAMMPS real or
  metal). Only the two systems listed above are recognised.
- **`units` selector for wall-clock timings.** The `.timings` file's
  stage durations are wall-clock measurements, not simulation
  quantities, and are always recorded in SI milliseconds.
- **Cross-CODATA-vintage matching.** The atomic-unit factors track the
  CODATA release shipped by the QCElemental version used to generate
  `atomic_constants.rs`. Reproducing a trajectory across hosts that
  use different generated files is not guaranteed even when both pick
  `units = "atomic"`.
