# Feature: Extended-XYZ Initial-State File <!-- rq-ff8e283d -->

A simulation's initial particle state is described by an extended-XYZ file
referenced from the TOML config (`config-schema.md`). The file carries the
particle count, the simulation box, per-particle type names, positions, and
optionally per-particle velocities. The runner consumes the parser's output to
construct a `ParticleState`, a `SimulationBox`, and a per-particle type-index
array used downstream by the force pipeline.

The format follows the extended-XYZ convention used by ASE, OVITO, and VMD,
restricted to the subset this project requires.

## File Format <!-- rq-cdd5dc6d -->

```
N
key1=value1 key2="value with spaces" ...
<row 1>
<row 2>
...
<row N>
```

- Line 1: a non-negative integer `N`, the particle count. Leading and trailing
  whitespace are tolerated.
- Line 2: the **comment line**, a space-separated list of `key=value`
  attributes. The supported attributes are described below. Unknown attributes
  are ignored. Values may be quoted with double quotes; quoted values may
  contain spaces.
- Lines 3 .. N+2: data rows, one per particle, with columns determined by the
  `Properties` attribute.

The file is encoded in UTF-8. Lines end in `\n` or `\r\n`. Lines after row N+2
must be either absent or blank (containing only whitespace); any non-blank
trailing content is an error.

### Comment-line attributes <!-- rq-20e7cab6 -->

#### `Lattice` (required) <!-- rq-9864078f -->

A nine-element space-separated list of `f64` values inside double quotes
representing the three Cartesian box vectors in row-major order (row 1 =
`a` vector, row 2 = `b` vector, row 3 = `c` vector):

```
Lattice="lx 0 0 xy ly 0 xz yz lz"
```

The parser accepts only lower-triangular boxes: the three "upper-
triangular" entries (positions 2, 3, 6 in the row-major list — `a_y`,
`a_z`, `b_z`) must be exactly `0.0`. The remaining six entries become
the six lattice parameters of the `SimulationBox`:

- Diagonal entries (positions 1, 5, 9 — `lx = a_x`, `ly = b_y`,
  `lz = c_z`) must be finite and strictly positive.
- Tilt entries (positions 4, 7, 8 — `xy = b_x`, `xz = c_x`,
  `yz = c_y`) must be finite; any sign and magnitude are accepted.

The orthorhombic case corresponds to `Lattice="lx 0 0 0 ly 0 0 0 lz"`
(all three tilts zero).

#### `Properties` (required) <!-- rq-3d3679d0 -->

A colon-separated specification of the data-column layout. The accepted
forms enumerate the four combinations of optional `velo` and optional
`image` columns, in this fixed order:

```
Properties=species:S:1:pos:R:3
Properties=species:S:1:pos:R:3:image:I:3
Properties=species:S:1:pos:R:3:velo:R:3
Properties=species:S:1:pos:R:3:velo:R:3:image:I:3
```

- `species:S:1` — one string column for the particle type name.
- `pos:R:3` — three real columns for the position (x, y, z) in metres.
- `velo:R:3` — optional three real columns for velocity (vx, vy, vz) in m/s.
- `image:I:3` — optional three integer columns for the image triple
  `(n_a, n_b, n_c)`, one per lattice direction. The unwrapped position
  is `pos + n_a · a + n_b · b + n_c · c` (see `particle-state.md`).
  For an orthorhombic box this reduces to
  `pos + image · (lx, ly, lz)`. When absent, image flags default to
  zero on load.

Any other ordering, missing required columns, or unexpected columns is an
error. The `velo` and `image` decisions are each all-or-nothing: when
declared, every data row must supply the three corresponding columns;
when absent, no row may supply them. `velo` and `image` are independent
(any combination of presence/absence is accepted as long as the order
shown above is followed).

Other extended-XYZ column types (`L` logical) and arbitrary property
names are not accepted in schema v1.

#### Other attributes <!-- rq-41eb5e02 -->

Any attribute not listed above is parsed but ignored (e.g. `Time=0.0`,
`Origin="0 0 0"`, comments embedded as `Comment="..."`). This preserves
forward compatibility with files generated by external tools.

### Data rows <!-- rq-d8a08c7a -->

Each row contains the columns declared by `Properties`, separated by one or
more whitespace characters. The number of columns must match the
`Properties` spec exactly.

- `species` — a string token (no whitespace inside). Must be a name declared
  in the consumer-supplied type list (see *Feature API*).
- `pos` — three `f64` values in metres. Each particle must lie in the
  primary image of the simulation box: its fractional coordinates
  (computed via `SimulationBox::fractional_coords`) must satisfy
  `s_a, s_b, s_c ∈ [-1/2, 1/2)`. Positions whose fractional component
  equals `+1/2` along any direction are rejected (the primary cell's
  upper bound is exclusive). For an orthorhombic box this reduces to
  `pos_x ∈ [-lx/2, lx/2)` etc.
- `velo` (when present) — three `f64` values in m/s. No range restriction.
- `image` (when present) — three `i32` values. No range restriction
  beyond the type bounds; negative and positive values are both accepted.

Non-finite values (NaN, infinity) in any real column are rejected;
integer columns that fail to parse as `i32` (out of range, non-integer
text) are rejected.

### Particle IDs <!-- rq-bc442f5b -->

The init file does not carry explicit particle IDs. Particle IDs are
assigned implicitly by data-row order: row 1 (the first data row) is
particle `0`, row 2 is particle `1`, and so on through `N-1`. This matches
`ParticleState::new`'s `ids=None` default (see `particle-state.md`).

### Empty file <!-- rq-484ef36b -->

`N = 0` is permitted. The file then contains only line 1 (`0`) and line 2
(the comment line with `Lattice` and `Properties`). No data rows follow.
This produces an empty `ParticleState` and a valid `SimulationBox`.

## Feature API <!-- rq-321fe0d0 -->

### Types <!-- rq-a953fc1d -->

- `InitState` — the parser's output. Fields: <!-- rq-8df7fb0c -->
  - `box: SimulationBox` — simulation box parsed from `Lattice` (in
    lower-triangular form; orthorhombic when all three tilts are zero).
  - `particle_count: usize` — the integer from line 1.
  - `type_indices: Vec<u32>` — for each particle, the zero-based index into
    the supplied `type_names` slice. Length equals `particle_count`.
  - `positions_x: Vec<f32>` — positions cast from `f64` to `f32` after
    validation. Length equals `particle_count`.
  - `positions_y: Vec<f32>`
  - `positions_z: Vec<f32>`
  - `velocities: Option<InitVelocities>` — `Some(_)` when `Properties`
    declares `velo:R:3`; `None` otherwise.
  - `images: Option<InitImages>` — `Some(_)` when `Properties` declares
    `image:I:3`; `None` otherwise. When `None`, the runner default-
    initialises the particle state's image flags to zero.

- `InitVelocities` <!-- rq-abd761d4 -->
  - `velocities_x: Vec<f32>` — velocities cast from `f64` to `f32`.
  - `velocities_y: Vec<f32>`
  - `velocities_z: Vec<f32>`

- `InitImages` <!-- rq-af0518f8 -->
  - `images_x: Vec<i32>` — image triple x-component. Length equals
    `particle_count`.
  - `images_y: Vec<i32>`
  - `images_z: Vec<i32>`

- `InitStateError` — error type returned by the parser. Variants: <!-- rq-573b650b -->
  - `Io(String)` — failed to read the file.
  - `Empty` — the file is empty (no lines).
  - `InvalidParticleCount { line_number: usize, raw: String }` — line 1 is
    missing, not an integer, or negative.
  - `MissingCommentLine` — the file has fewer than two lines.
  - `MissingAttribute { name: &'static str }` — required `Lattice` or
    `Properties` attribute absent.
  - `InvalidLattice(String)` — `Lattice` is malformed, has a non-zero
    upper-triangular entry (one of `a_y`, `a_z`, `b_z`), contains
    non-finite components, or has a non-positive diagonal.
  - `InvalidProperties(String)` — `Properties` does not match one of the
    four accepted forms.
  - `RowCountMismatch { expected: usize, actual: usize }` — the number of
    non-blank data rows differs from `N`.
  - `RowColumnCountMismatch { line_number: usize, expected: usize, actual: usize }`
    — a data row has the wrong number of whitespace-separated columns.
  - `UnknownType { line_number: usize, name: String }` — `species` value is
    not present in the caller-supplied `type_names` slice.
  - `InvalidNumber { line_number: usize, column: &'static str, raw: String }`
    — a real column could not be parsed as `f64`, or an integer column
    could not be parsed as `i32`.
  - `NonFiniteValue { line_number: usize, column: &'static str }` — a
    numeric column is NaN or infinity.
  - `PositionOutsideBox { line_number: usize, direction: &'static str, fractional: f64 }`
    — a position falls outside the primary image of the simulation box.
    `direction` is one of `"a"`, `"b"`, `"c"`; `fractional` is the
    out-of-range fractional coordinate (either `>= 0.5` or `< -0.5`).
  - `TrailingContent { line_number: usize }` — non-blank content beyond row
    `N+2`.

### Functions <!-- rq-dad38fdd -->

- `load_init_state(path: &Path, type_names: &[&str]) -> Result<InitState, InitStateError>` <!-- rq-5711e6b2 -->
  - Reads the file at `path` and parses it according to *File Format*.
  - `type_names` is the list of declared type names from the config,
    indexed left-to-right. The `species` value on each data row must equal
    one of these names; the parser stores the corresponding zero-based
    index in `type_indices`.
  - Validates every field as described, including position-in-box checks
    using the parsed `Lattice` dimensions.
  - Returns the populated `InitState` on success.

The function reads the file once, in a single linear pass. Validation
ordering: first line 1, then line 2 attributes (`Lattice` before
`Properties`), then each data row in source order.

## Out of Scope <!-- rq-6897be01 -->

- Crystallographic `a, b, c, α, β, γ` lattice syntax; the parser
  accepts only the row-major 9-component lattice matrix described
  above. Converting from `a, b, c, α, β, γ` is the caller's
  responsibility.
- Property columns other than `species:S:1`, `pos:R:3`, `velo:R:3`. Forces,
  masses, charges, type-as-integer columns, etc. are all rejected in schema
  v1.
- Multi-frame init files. Only the first frame is read; in schema v1 the
  parser rejects any non-blank content after row `N+2`.
- Per-particle masses in the init file. Masses come from the config's
  `[[particle_types]]`.
- Restart files (a separate planned feature).
- Big-endian / little-endian binary variants.
- Compressed (gzip, bzip2, xz) input files.
- Numerical validation of velocities (NaN/Inf in `pos` is rejected; in
  `velo` is also rejected for symmetry).
- Silent wrapping of out-of-cell positions; positions must already lie in
  the primary cell.
